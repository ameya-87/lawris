## 1. Executive Summary

**Lawris** is an AI-native case-management agent for Indian advocates. It collapses four broken workflows into one tab:

1. **Case management** — clients, cases, documents, hearings
2. **Statutory deadline intelligence** — auto-computes the BNSS s.187(3) chargesheet rule, Limitation Act periods, and hearing schedules from raw FIR data
3. **AI document drafting** — court-ready bail applications and plaints in <30 seconds, using live case context and Indian-court formatting
4. **Case-grounded legal research** — questions answered against the specific case's facts, with structured citations

**Stack one-liner:** Next.js 14 App Router + Supabase Postgres + Google Gemini 2.5 Flash, deployed as a single Vercel app.

**Why it wins:** No competitor (Clio, MyKase, Harvey AI, IndianKanoon) ships all four columns. Lawris is the first product in the Indian legal-tech market where case management, AI drafting, deadline intelligence, and grounded research are unified.

**Status:** Working end-to-end prototype. Live URL: `http://10.11.11.182:4000` (LAN) or VS Code-tunneled `localhost:4000`. Live database in Supabase project `rxqhbmtvgdgkubnetpsy`.

## 2. Problem Statement

| Pain | Stat (from market research) |
|---|---|
| Lost time locating documents | 6+ hrs / week / lawyer |
| Time spent on repetitive drafting | 40–60% of billable hours |
| Lawyers using email + Word as case manager | 77% |
| Malpractice claims traceable to calendaring failures | ~25% |
| Active matters per Indian advocate | 50–200 |

**Why nobody has solved this:** US/global tools (Clio, Harvey) don't speak Indian law. Indian tools (MyKase, ProVakil, IndianKanoon) lack AI. The BNSS replacing CrPC in July 2024 voided every existing template library — a market-timing window opened.

**Stakes:** Missing a 60/90-day chargesheet deadline lets the accused walk on default bail. A missed limitation date extinguishes the right to sue forever. These are not productivity issues — they are liberty and livelihood issues.

## 3. Solution Overview — The Four Pillars

### 3.1 Deadline Brain
- **Input:** FIR date, max sentence years, case type
- **Output:** Auto-generated `deadlines` rows for chargesheet (60-day if <10 yrs, 90-day if ≥10 yrs per BNSS s.187(3)), limitation periods, hearing dates
- **UI surfacing:** Color-coded urgency pills (red ≤3d / amber ≤7d / yellow ≤30d / green) on dashboard, case detail, and a global calendar
- **Implementation:** [src/lib/deadlines.ts](vit_hackathon/src/lib/deadlines.ts) (pure functions) → invoked from `POST /api/cases` ([src/app/api/cases/route.ts](vit_hackathon/src/app/api/cases/route.ts))

### 3.2 AI Document Drafting
- **Two doc types:** Regular bail application (Sec 483 BNSS / Sec 439 CrPC), Civil plaint (Order VII CPC)
- **Streaming:** Tokens stream live from Gemini → reach the editor pane in real time
- **Persistence:** Final draft saved to `documents` table on stream completion
- **Implementation:** [src/app/api/ai/draft/route.ts](vit_hackathon/src/app/api/ai/draft/route.ts) → [src/lib/prompts/bail-application.ts](vit_hackathon/src/lib/prompts/bail-application.ts) and [src/lib/prompts/plaint.ts](vit_hackathon/src/lib/prompts/plaint.ts)

### 3.3 Case-Grounded Legal Research
- **Grounded in current case context** (case type + AI summary) — not generic Q&A
- **Structured JSON output:** `{ answer, citations[], statutes[], follow_ups[] }`
- **Implementation:** [src/app/api/ai/research/route.ts](vit_hackathon/src/app/api/ai/research/route.ts) → [src/lib/prompts/research.ts](vit_hackathon/src/lib/prompts/research.ts) (uses Gemini `responseSchema` for forced structure)

### 3.4 Hearing Memory & Auto-Summary
- **After every hearing log:** the case's `ai_summary` field is regenerated by Gemini against the full hearing history
- **Side effect:** if `next_date` is provided, a new `hearing` deadline is auto-inserted
- **Implementation:** [src/app/api/cases/[id]/hearings/route.ts](vit_hackathon/src/app/api/cases/[id]/hearings/route.ts) → [src/lib/prompts/summarise.ts](vit_hackathon/src/lib/prompts/summarise.ts)

**Key architectural decisions:**
- **No separate backend service.** Everything is Next.js API routes for blast radius and deploy simplicity.
- **Service-role Supabase key, no RLS.** Single-user MVP; `LAWYER_ID` env var pins all queries to the seeded lawyer. RLS is the obvious post-MVP hardening step.
- **Streaming AI via raw `ReadableStream`**, not Server-Sent Events or WebSockets. Avoids extra protocol surface, works through any HTTP proxy.
- **Server components do the data fetching** (Next.js 14 RSC pattern); client components are reserved for interactivity (forms, AI panels, tabs).

## 7. API Surface

All routes are Next.js App Router handlers. All server-side. Auth: implicit single-user via `LAWYER_ID` env var.

| Method | Path | Purpose | Body | Returns |
|---|---|---|---|---|
| GET | `/api/health` | Wiring smoke test | — | `{ ok, env: {...}, timestamp }` |
| GET | `/api/cases` | List all cases for lawyer | — | `{ cases: Case[] }` |
| POST | `/api/cases` | Create case + auto-deadlines | `{ title, case_type, fir_date?, offence_max_years?, ... }` | `{ case, auto_deadlines: number }` |
| GET | `/api/cases/:id` | Full case bundle | — | `{ case, deadlines, documents, hearings, research }` |
| POST | `/api/cases/:id/hearings` | Log hearing + auto-summarise | `{ hearing_date, what_happened, next_date?, ... }` | `{ hearing }` (side effects: regenerates `cases.ai_summary`, may insert deadline) |
| GET | `/api/deadlines/upcoming` | Cross-case 60-day window | — | `{ deadlines: (Deadline & { cases: ... })[] }` |
| PUT | `/api/deadlines/:id/complete` | Mark deadline done | — | `{ ok: true }` |
| **POST** | **`/api/ai/draft`** | **Stream AI draft (centerpiece)** | `{ case_id, doc_type: "bail_app" \| "plaint", extra? }` | **`text/plain` stream**; persists to `documents` on completion |
| POST | `/api/ai/research` | Case-grounded Q&A | `{ case_id, query }` | `{ answer, citations[], statutes[], follow_ups[] }` (also persists to `research_notes`) |

All POST routes validate input with **zod**. Validation failures return 400 with `error.format()`.

## 8. AI Layer — Prompt Architecture

Each Gemini call uses a separate **system instruction** (rules, format) + **user content** (case data). This separation makes prompts portable to any other LLM and easy to edit without breaking call sites.

### 8.1 Prompt files
| File | Used by | Output format |
|---|---|---|
| [src/lib/prompts/bail-application.ts](vit_hackathon/src/lib/prompts/bail-application.ts) | `/api/ai/draft` (`doc_type=bail_app`) | Markdown body of bail app, no preamble |
| [src/lib/prompts/plaint.ts](vit_hackathon/src/lib/prompts/plaint.ts) | `/api/ai/draft` (`doc_type=plaint`) | Markdown body of plaint, no preamble |
| [src/lib/prompts/research.ts](vit_hackathon/src/lib/prompts/research.ts) | `/api/ai/research` | Strict JSON: `{ answer, citations, statutes, follow_ups }` |
| [src/lib/prompts/summarise.ts](vit_hackathon/src/lib/prompts/summarise.ts) | `/api/cases/:id/hearings` (post-insert) | 4–8 sentence prose case brief |

### 8.2 Gemini wrapper
[src/lib/gemini.ts](vit_hackathon/src/lib/gemini.ts) exports:
- `gemini()` — singleton `GoogleGenAI` client
- `streamDraft({systemInstruction, userPrompt})` — `AsyncGenerator<DraftStreamChunk>` — used by streaming routes
- `generateText({systemInstruction, userPrompt, temperature, responseSchema?})` — single-shot JSON or text — used by research and summarise

### 8.3 What each prompt enforces
- **Bail application:** Section 483 BNSS / 439 CrPC framing, Article 21 reference, case-law anchors (*Satender Kumar Antil*, *Dataram Singh*, *Balchand*), standard "MOST RESPECTFULLY SHOWETH" structure, prayer clause, advocate sign-off block. Adapts to POCSO/NDPS/UAPA where flagged.
- **Plaint:** Order VII CPC compliance, jurisdiction + limitation + court fee paragraphs, verification block.
- **Research:** Real precedents only ("OMIT if not confident — do not fabricate"), at least one Supreme Court citation, BNSS/BNS + CrPC/IPC cross-reference where relevant.
- **Summarise:** 4–8 sentences, no bullet points, no citations, no preamble — pure prose brief.

### 8.4 Quality-tested outputs
Verified live against seeded data:
- Bail app for the POCSO case correctly cites all three precedents, pulls FIR `0289/2026`, accused name `Rahul Sharma`, arrest date `2026-01-23`, and *spotted on its own* that the chargesheet was overdue.
- Research on "POCSO bail grounds" mentioned the 88-day chargesheet delay (case-specific) and cited Sushila Aggarwal + Hassan Ali Khan.
- Summarise after a logged hearing produced a 5-sentence brief that named the *Satender Kumar Antil* argument I'd just typed.

## 10. End-to-End Demo Path (call traces)

### 10.1 "Open the POCSO case → see the critical deadline"
```
Browser GET /cases/aaaaaaaa-…
  → src/app/cases/[id]/page.tsx (server component)
    → loadCase() (parallel Supabase queries: case, deadlines, documents, hearings, research)
      → Supabase REST (service-role, filtered by lawyer_id)
    → renders <CaseDetail>
      → src/components/case-detail.tsx (client)
        → tab "Deadlines" → <DeadlineList>
          → src/components/deadline-list.tsx
          → classifyUrgency(due_date) from src/lib/deadlines.ts
          → "in 2d" pill rendered red via urgencyClass()
```

### 10.2 "Click 'Draft Bail Application' → watch it stream"
```
User clicks button in <AiDraftPanel>
  → fetch POST /api/ai/draft  { case_id, doc_type: "bail_app" }
    → src/app/api/ai/draft/route.ts
      → zod parse body
      → Supabase: load case + client + lawyer rows
      → bailApplicationUserPrompt(...) — JSON-stringify case data
      → streamDraft() → gemini.models.generateContentStream({...})
        → for-await chunks → encode → ReadableStream controller.enqueue
      → on stream end: insert into documents (source: ai_drafted, content: collected)
  → Browser reads stream chunk-by-chunk
    → setDraft(d => d + decodedText) — re-renders editor pane each tick
    → mdToHtml() converts incremental Markdown to HTML
```

### 10.3 "Add a hearing → AI summary updates"
```
User submits hearing form in <HearingPanel>
  → fetch POST /api/cases/:id/hearings
    → zod parse
    → Supabase insert hearing_logs row
    → if next_date: Supabase insert deadlines row (auto)
    → Supabase fetch case + all hearings
    → generateText() with summarise prompt → 5-sentence brief
    → Supabase update cases.ai_summary
  → router.refresh() in client → server component re-fetches → header re-renders with new summary
```

### 10.4 "Ask research → get cited answer"
```
User types question in <ResearchPanel>
  → fetch POST /api/ai/research { case_id, query }
    → Supabase load case (for ai_summary + case_type context)
    → generateText() with research prompt + responseSchema (JSON forced)
    → JSON.parse the response → ResearchResponse
    → Supabase insert research_notes (Q + A + citations stored)
  → Browser renders structured turn (answer + cited cases + statutes + follow-ups)
```

## 16. Operational Runbook (when something breaks)

| Symptom | Diagnosis | Fix |
|---|---|---|
| Dashboard shows "0 critical · 0 high" | Seed dates are stale — `current_date - X days` in old run | Re-run `scripts/seed.sql` to re-anchor dates |
| `/api/health` returns 500 | Env var missing | Check `.env.local`; restart dev server (`pkill -f "next dev"` → re-launch) |
| `/api/ai/draft` errors instantly | Gemini API key invalid or rate-limited | Verify `GEMINI_API_KEY`; check AI Studio dashboard for quota |
| Stream cuts off mid-document | Network blip; rare | Click Draft again — second attempt usually works |
| `next build` fails with disk error | VM disk full | `rm -rf ~/.cache/go-build && npm cache clean --force` (frees ~800 MB) |
| Browser can't reach `localhost:4000` from laptop | VS Code tunnel broken | Use `http://10.11.11.182:4000` directly (same-LAN) |
| Dev server SIGTERMs randomly | Background-task lifecycle | Use `nohup env PORT=4000 npm run dev > /tmp/lawris-dev.log 2>&1 &; disown` |
